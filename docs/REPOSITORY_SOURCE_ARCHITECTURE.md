# Архитектура получения репозитория для Blazor Server MVP

## Анализ вариантов

### 1. Ручной ввод пути
- Преимущества: простота реализации, работает локально, подходит для раннего MVP.
- Недостатки: зависит от локального файлового пути, плохо переносится на сервер и корпоративную среду, неудобен для GitHub/GitLab.
- Ограничения браузера: браузер не может безопасно получить доступ к произвольной файловой системе клиента; в серверном режиме это требует серверной обработки и обычно не подходит для обычного веб-сценария.
- Локально: да, но только как локальная рабочая схема.
- После публикации: ограниченно и небезопасно.
- Вписывание в архитектуру: слабое, потому что привязывает бизнес-логику к конкретному способу получения данных.

### 2. Выбор локальной папки
- Преимущества: удобнее, чем строка, и лучше соответствует локальному сценарии.
- Недостатки: всё ещё требует доступа к файловой системе на стороне сервера/клиента и не переносится напрямую на серверную публикацию.
- Ограничения браузера: стандартный браузер не даёт обычному веб-приложению полный доступ к файловой системе клиента без специального ввода/выбора файла.
- Локально: да, особенно при запуске Blazor Server на машине разработчика.
- После публикации: возможно только с серверной файловой системой или специальной инфраструктурой.
- Вписывание в архитектуру: лучше, чем ручный путь, но всё ещё не универсально.

### 3. Загрузка ZIP-архива
- Преимущества: переносим, работает через браузер, подходит для серверной публикации, не зависит от локальной файловой системы.
- Недостатки: требует распаковки, не даёт прямого доступа к Git-истории, сложнее для полноценного анализа diff.
- Ограничения браузера: поддерживается без проблем через обычный upload.
- Локально: да.
- После публикации: да.
- Вписывание в архитектуру: хорошо, но для MVP не лучший вариант, если нужен Git-дифф и история коммитов.

### 4. Клонирование по Git URL
- Преимущества: универсальный сценарий для локальной и серверной среды, хорошо подходит для GitHub/GitLab, логически близок к будущей модели использования.
- Недостатки: требует сетевого доступа, аутентификации и возможностей клонирования, больше операций по инфраструктуре.
- Ограничения браузера: браузер сам по себе не должен клонировать Git; это выполняется на сервере.
- Локально: да.
- После публикации: да, если есть доступ к сети и правам на клонирование.
- Вписывание в архитектуру: лучше всего, потому что не привязывает бизнес-логику к конкретному источнику данных.

### 5. Другие варианты
- Подключение к GitHub/GitLab API, локальные git-объекты, file share, внутренний репозиторий, встроенный агент. 
- Все они подходят как последующие расширения, но для MVP требуют дополнительного уровня инфраструктуры.

## Выбор для MVP

Лучший вариант для MVP — получение репозитория через абстракцию источника и поддержку локального пути/локальной папки в качестве первого рабочего сценария.

Почему именно так:
- архитектура уже не зависит от строки с локальным путём;
- бизнес-логика работает через единый контракт IRepositoryProvider;
- впоследствии можно добавить ZIP, Git URL и другие источники без изменения логики анализа;
- это даёт устойчивую основу для перехода от локального запуска к серверному и корпоративному использованию.

## Как реализовано

- UI принимает источник репозитория и его значение;
- приложение преобразует ввод в RepositoryRequest;
- IRepositoryProvider отвечает за получение реального пути к Git-репозиторию;
- GitAnalyzer работает только с уже полученным репозиторием и не знает, как он был получен.
