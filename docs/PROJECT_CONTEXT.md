# PROJECT_CONTEXT

## 1. Executive Summary
Система представляет собой внутренний MVP для автоматизированного анализа изменений в локальном Git-репозитории с последующей генерацией структурированного отчёта для web-интерфейса. Цель документа — зафиксировать архитектурные границы, основные технологические решения и ограничения MVP до начала активной разработки.

Архитектура ориентирована на одиночную команду разработки и короткий горизонт реализации. По этой причине выбран подход Clean Architecture Lite: он сохраняет разделение ответственности, но не перегружает проект избыточной абстракцией и дополнительными уровнями, которые не дают ценности на старте.

| Параметр | Выбор |
|---|---|
| Backend | ASP.NET Core 9 |
| Frontend | Blazor Server |
| База данных | PostgreSQL |
| ORM | Entity Framework Core (Code First) |
| Git-интеграция | LibGit2Sharp |
| Анализ C# | Roslyn |
| Архитектурный стиль | Clean Architecture Lite |
| Доступ к данным | Repository Pattern |
| Композиция зависимостей | Dependency Injection |

Почему выбран именно такой набор: он позволяет быстро собрать рабочий MVP на одном языке и одном runtime, не вводя лишней распределённости и не усложняя сопровождение. Альтернативы в виде микросервисов, отдельного SPA-фронтенда или оркестрации через очередь задач были бы избыточны для текущего объёма, увеличили бы стоимость поддержки и замедлили бы поставку.

Преимущества: единый стек C#, хорошая тестируемость, предсказуемая эволюция, низкий порог входа. Недостатки: более высокая связность внутри одного deployable unit, зависимость от производительности одного приложения и более ограниченная горизонтальная масштабируемость на MVP.

## 2. Project Goals
Цель проекта — сократить время первичного code review за счёт автоматического анализа Diff, статического анализа и формирования структурированного отчёта. Система должна не заменять инженера, а снижать количество рутинной ручной работы и повышать качество первичной проверки изменений.

Проект решает следующие задачи:

| Цель | Описание |
|---|---|
| Ускорение ревью | Автоматическое выделение проблемных зон до ручного просмотра |
| Повышение качества | Единый pipeline анализа снижает вероятность пропуска дефектов |
| Воспроизводимость | Результат анализа должен быть повторяемым для одной и той же версии кода |
| Расширяемость | Архитектура должна допускать подключение новых языков и новых типов анализаторов |
| Подготовка к будущему AI | Модель данных и pipeline должны быть готовы к отдельному AI-модулю |

Почему выбран именно такой подход: система строится вокруг конкретного сценария локального анализа репозитория, а не вокруг универсальной платформы разработки. Для MVP выгоднее сначала решить одну задачу хорошо, чем сразу строить универсальный продукт с большим числом интеграций.

Альтернативы и причины отказа:

| Альтернатива | Почему не подходит для MVP |
|---|---|
| Полноценная платформа с несколькими источниками кода | Увеличивает объём реализации и не нужна для первого релиза |
| Сразу AI-first архитектура | Создаёт зависимость от LLM до того, как стабильна база анализа |
| Интеграция в GitHub/GitLab на старте | Усложняет безопасность, авторизацию и коммуникацию с внешними API |

Преимущества выбранного подхода: ясный scope, быстрый результат, контролируемый архитектурный долг. Недостатки: ограниченная внешняя интеграция в первом релизе и отсутствие распределённой обработки.

## 3. Functional Requirements
Ниже приведён целевой функциональный состав MVP.

| № | Требование |
|---|---|
| 1 | Пользователь указывает путь к локальному Git-репозиторию |
| 2 | Система валидирует доступность репозитория и наличие Git-метаданных |
| 3 | Система определяет Diff и изменённые файлы |
| 4 | Система идентифицирует тип изменения файла |
| 5 | Система анализирует только C#-файлы на уровне MVP |
| 6 | Система выполняет статический анализ найденного C#-кода |
| 7 | Система сохраняет результаты анализа в PostgreSQL |
| 8 | Система отображает результаты в Web UI |
| 9 | Система формирует Markdown-отчёт |
| 10 | Система хранит историю запусков анализа |

Расширенные функциональные требования, заложенные в архитектуру, но не полностью реализуемые на MVP:

| Требование | Статус |
|---|---|
| Поддержка новых языков через ILanguageAnalyzer | Заложено в архитектуру |
| Поддержка AI Review | Отложено в будущий модуль |
| CLI-запуск | Отложено |
| Git Hooks | Отложено |
| Множественные LLM-провайдеры | Отложено |

Почему именно такой набор: MVP должен быть завершённым и полезным, но не обязан покрывать все возможные сценарии использования. Альтернативой было бы включение большого числа функций сразу, однако это ухудшило бы качество базового сценария и замедлило бы разработку.

Преимущества: понятный объём, простая проверка готовности, лёгкая приоритизация задач. Недостатки: часть целевой платформы будет существовать только как архитектурная заготовка.

## 4. Non Functional Requirements
Нефункциональные требования фиксируют свойства, без которых система не сможет считаться пригодной для внутреннего использования.

| Категория | Требование |
|---|---|
| Производительность | Обработка малого Diff должна укладываться в приемлемое время для интерактивного сценария |
| Тестируемость | Ключевая логика должна быть отделена от инфраструктуры |
| Поддерживаемость | Изменения в одном слое не должны разрушать другие слои |
| Безопасность | Локальные пути и контекст должны обрабатываться безопасно |
| Масштабируемость | Архитектура должна допускать развитие без полной переработки |
| Расширяемость | Должна быть возможность подключать новые анализаторы и языки |

Почему выбран именно такой набор: для внутреннего MVP важнее предсказуемость и поддерживаемость, чем экстремальная пропускная способность. Альтернативы в виде асинхронной распределённой обработки или отдельного вычислительного кластера не оправданы текущим объёмом проекта.

Преимущества: система проще в отладке, быстрее в поставке и дешевле в сопровождении. Недостатки: ограничение на рост throughput и зависимость от производительности одного сервиса.

## 5. Architecture Overview
Выбрана Clean Architecture Lite с логическим разделением на Domain, Application, Infrastructure и Web. Стиль сознательно упрощён относительно классической Clean Architecture: сохранены только те уровни, которые реально дают пользу на MVP.

ASCII-схема общего устройства:

```text
Browser
	|
	v
Blazor Server UI
	|
	v
Application Layer
	|
	v
Infrastructure Layer
	|       |        |
	|       |        +--> PostgreSQL
	|       +--> Roslyn / LibGit2Sharp
	+--> File/Report Services
```

Почему выбран именно этот подход: он позволяет удержать бизнес-логику в центре системы и минимизировать зависимость от внешних технологий. Альтернативой был бы более жёсткий DDD-стек с большим количеством агрегатов и доменных событий, но для двухнедельного MVP это создаст лишнюю сложность без пропорциональной пользы.

Преимущества: изоляция, тестируемость, ясные границы ответственности, возможность развития в сторону CLI и AI-модуля. Недостатки: требуется дисциплина при соблюдении границ, а неправильное упрощение может привести к “расплыванию” ответственности между слоями.

## 6. Static Analysis Pipeline
Конвейер анализа построен как последовательность технологических стадий, каждая из которых обогащает контекст и снижает шум для следующей стадии.

```text
Git Repository
	|
	v
Git Analyzer
	|
	v
Определение изменённых файлов
	|
	v
ILanguageAnalyzer
	|
	v
RoslynLanguageAnalyzer (C#)
	|
	v
Static Analysis
	|
	v
Report Generator
	|
	v
Web UI
```

Почему выбран именно такой порядок: сначала система должна точно понять, что изменилось, затем определить семантический контекст, после этого применить правила статического анализа и только потом отдать результат пользователю. Если пропустить ранние стадии, отчёт станет менее точным и менее полезным.

Альтернативы:

| Альтернатива | Почему не подходит |
|---|---|
| Прямой анализ Diff без семантики | Потеря контекста и рост ложных срабатываний |
| Анализ только по тексту файлов | Не позволяет понимать связи между символами |
| Сразу AI-оценка без статического анализа | Сильно ухудшает качество и воспроизводимость |

Преимущества pipeline: поэтапное обогащение контекста, масштабируемость, возможность заменять отдельные стадии. Недостатки: увеличивается суммарная стоимость анализа и сложность оркестрации.

## 7. Language Support
На MVP поддерживается только C# через RoslynLanguageAnalyzer. Архитектура, однако, проектируется так, чтобы новые языки могли быть подключены без переписывания основного pipeline.

| Язык | Статус | Примечание |
|---|---|---|
| C# | MVP | Реализуется через Roslyn |
| Другие языки | Future | Подключение через ILanguageAnalyzer |

Почему выбрано именно так: основной стек проекта — .NET и C#, поэтому поддержка C# даёт максимальную отдачу от первой версии. Альтернативой было бы распыление на несколько языков сразу, но это увеличило бы объём анализа и усложнило бы проектирование.

Преимущества: концентрация на одном языке, высокая точность анализа, использование нативных средств платформы. Недостатки: в MVP система не анализирует многозначные polyglot-репозитории полноценно.

Архитектурный принцип расширения:

| Элемент | Назначение |
|---|---|
| ILanguageAnalyzer | Единый контракт для языковых анализаторов |
| RoslynLanguageAnalyzer | Специализация для C# |
| Language Analyzer Registry | Точка выбора анализатора по расширению или типу файла |

## 8. Solution Structure
Рекомендуемая структура solution должна быть минимальной, но достаточной для разделения ответственности.

```text
AIReviewSystem
├── AIReviewSystem.Domain
├── AIReviewSystem.Application
├── AIReviewSystem.Infrastructure
├── AIReviewSystem.Web
├── AIReviewSystem.Contracts
├── AIReviewSystem.Tests.Unit
└── AIReviewSystem.Tests.Integration
```

| Проект | Назначение |
|---|---|
| Domain | Предметная модель и бизнес-правила |
| Application | Сценарии использования и оркестрация |
| Infrastructure | Технические адаптеры и интеграции |
| Web | Пользовательский интерфейс и composition root |
| Contracts | Стабильные DTO и форматы обмена |
| Tests.Unit | Модульные тесты домена и application |
| Tests.Integration | Проверка интеграции с БД, Git и анализаторами |

Почему выбран именно такой набор: он достаточен для MVP и не создаёт избыточной фрагментации. Альтернативой было бы смешать всё в одном проекте, но это быстро привело бы к техническому долгу и усложнило бы тестирование.

Преимущества: понятная структура, удобная навигация, разделение зависимостей. Недостатки: для небольшого MVP это чуть больше файлов и проектов, чем в монолите.

## 9. Domain Layer
Domain Layer содержит только то, что описывает предметную область и её инварианты. Внутри него не должно быть привязки к PostgreSQL, Blazor, Roslyn, LibGit2Sharp или внешним сервисам.

| Сущность / концепт | Ответственность |
|---|---|
| AnalysisSession | Единица анализа и верхнеуровневый статус |
| RepositorySnapshot | Зафиксированное состояние репозитория в момент анализа |
| ChangedFile | Модель изменённого файла |
| StaticFinding | Нормализованное замечание статического анализа |
| ReportArtifact | Результат экспорта отчёта |
| Value Objects | Коммит-диапазоны, путь, позиция, severity |

Почему выбран именно такой подход: предметная модель должна быть стабильной и независимой от техники реализации. Альтернативой был бы анемичный домен с переносом всей логики в сервисы, но это ухудшает выразительность модели и затрудняет поддержание правил предметной области.

Преимущества: чистота модели, простота тестирования, низкая связность. Недостатки: требуется дисциплина, чтобы не переносить инфраструктурную логику в домен.

---

## 10. Application Layer
Application Layer координирует сценарии использования и определяет, как доменная модель взаимодействует с внешними сервисами через абстракции.

| Подсистема | Роль |
|---|---|
| Use Cases | Запуск анализа, получение истории, экспорт отчёта |
| Orchestration | Пошаговое выполнение pipeline |
| DTO | Контракты для входа и выхода |
| Validators | Проверка корректности входных данных |
| Policies | Правила сортировки, ранжирования и агрегации |

Почему выбран именно такой подход: Application Layer позволяет держать сценарий анализа единообразным независимо от того, пришёл ли запрос из Web UI, будущего CLI или hook-механизма. Альтернативой был бы прямой вызов инфраструктуры из UI, но это быстро разрушает архитектурные границы.

Преимущества: управляемая оркестрация, хорошая тестируемость, изоляция use cases. Недостатки: требуется несколько дополнительных абстракций и согласованность между слоями.

---

## 11. Infrastructure Layer
Infrastructure Layer содержит все технические реализации, которые обслуживают бизнес-сценарий.

| Подсистема | Назначение |
|---|---|
| Persistence | Работа с PostgreSQL через EF Core |
| Git | Доступ к локальному Git-репозиторию через LibGit2Sharp |
| RoslynAnalysis | Семантический анализ C# |
| StaticAnalysis | Набор правил и нормализация diagnostics |
| Reporting | Генерация Markdown-отчётов |
| Configuration | Подключение настроек и параметров |

Почему выбран именно такой подход: все внешние зависимости вынесены на периферию, чтобы их можно было менять без изменения ядра системы. Альтернативой было бы смешение инфраструктуры с application, но это сильно осложнило бы модификации и тестирование.

Преимущества: сменяемость компонентов, прозрачные границы, удобство интеграционных тестов. Недостатки: больше точек интеграции и выше требования к качеству контрактов.

---

## 12. Web Layer
Web Layer реализован на Blazor Server и отвечает за пользовательский сценарий запуска анализа, отображения результатов и навигации по истории запусков.

| Элемент | Назначение |
|---|---|
| Pages | Основные экраны приложения |
| Components | Повторно используемые UI-блоки |
| State | Состояние текущего анализа и UI |
| Services | Вызовы application-сценариев |
| Middleware | Обработка ошибок и логирование |

Почему выбран именно Blazor Server: для внутреннего MVP это минимизирует количество технологий и позволяет быстро собирать интерфейс на C#. Альтернативой был бы отдельный SPA-фронтенд, но он добавил бы отдельный стек, усложнил бы обмен данными и увеличил бы сроки разработки.

Преимущества: единый язык, быстрый старт, тесная интеграция с backend. Недостатки: зависимость от серверного соединения, менее независимый frontend и возможные ограничения при высокой интерактивности.

## 13. Database Design
Для хранения аналитических сессий и результатов выбран PostgreSQL. Причина выбора — зрелая реляционная модель, хорошая работа с EF Core и достаточный набор возможностей для структурированных результатов анализа.

### Логическая схема

```text
AnalysisSession
	|-- ChangedFile
	|-- StaticFinding
	|-- ReportArtifact
	|-- RepositorySnapshot
```

### Основные таблицы

| Таблица | Что хранит |
|---|---|
| analysis_sessions | Метаданные запуска и статус |
| repository_snapshots | Точку сравнения и commit context |
| changed_files | Изменённые файлы и их признаки |
| static_findings | Результаты статического анализа |
| report_artifacts | Метаданные отчётов и экспортов |

Почему выбрана именно реляционная схема: данные анализа хорошо структурированы, связаны между собой и требуют надёжной целостности. Альтернативой была бы схема с хранением всего в JSON-документах, но это ухудшило бы фильтрацию, историю и поддержку отчётов.

Преимущества: транзакционность, согласованность, удобные выборки по истории. Недостатки: меньше гибкости для сверхслабоструктурированных данных.

## 14. Dependency Injection
Dependency Injection используется как основной механизм композиции системы и подмены реализаций.

| Тип жизненного цикла | Пример применения | Почему |
|---|---|---|
| Singleton | Конфигурации, фабрики, stateless helpers | Экономия ресурсов и единая конфигурация |
| Scoped | DbContext, репозитории, use case orchestration | Согласованность на уровне одного запроса/операции |
| Transient | Лёгкие сервисы обработки и маппинга | Нет необходимости в общем состоянии |

Почему выбран именно такой подход: DI позволяет сделать архитектуру заменяемой и хорошо тестируемой без жёстких связей между классами. Альтернативой был бы service locator или ручная инициализация зависимостей, но это делает систему менее прозрачной и хуже поддерживаемой.

Преимущества: централизованная композиция, тестируемость, гибкость подмены зависимостей. Недостатки: при плохой дисциплине можно перегрузить composition root и потерять читаемость.

## 15. Repository Pattern
Repository Pattern выбран для изоляции Application Layer от деталей хранения данных и для поддержки тестируемости без реальной БД.

| Задача | Причина использования репозитория |
|---|---|
| Чтение истории | Упрощённый доступ к сессиям и отчётам |
| Сохранение результатов | Контроль консистентности и транзакций |
| Модульное тестирование | Возможность подменить хранилище заглушками |

Почему выбран именно такой подход: для предметных сущностей и агрегатов полезно иметь ограниченный доступ к данным через понятный контракт. Альтернативой является прямое использование DbContext во всех слоях, но это делает приложение более связанным с EF Core и ухудшает переносимость логики.

Преимущества: ясные границы доступа, удобство тестирования, меньше привязки к ORM. Недостатки: риск формальной избыточности, если репозитории использовать там, где достаточно query-сервиса.

---

## 16. Static Analysis
Статический анализ на MVP применяется только к C# и ориентирован на выявление проблем, которые можно обнаружить без AI.

| Категория | Примеры проверок |
|---|---|
| Структурные | Неполные ссылки, некорректные зависимости |
| Качество кода | Потенциально опасные конструкции, избыточная сложность |
| Семантические | Использование символов и связей между файлами |
| Контекстные | Зависимости изменённого кода от окружающих элементов |

Почему выбран именно Roslyn: это нативный и наиболее точный инструмент для анализа C# в экосистеме .NET. Альтернативой могли бы быть regex-подходы, сторонние линтеры или парсинг текста, но они не дают достаточной семантики для этого проекта.

Преимущества: точный анализ C#, доступ к semantic model, высокая релевантность для .NET-кода. Недостатки: поддержка только C# на MVP и зависимость от корректности исходных проектов/решения.

## 17. AI Module (Future)
AI-модуль не входит в текущий MVP и рассматривается как отдельный будущий компонент. На этой стадии система должна полностью работать без AI и выдавать отчёт на основе Git, Roslyn и статического анализа.

Будущая схема взаимодействия:

```text
Static Analysis
	|
	v
Context Builder
	|
	v
AI Reviewer
	|
	v
Aggregator
	|
	v
Report
```

Почему AI вынесен отдельно: он должен быть подключаемым модулем, а не обязательной зависимостью базового сценария. Альтернативой была бы жёсткая интеграция AI в MVP, но это увеличило бы риски, зависимость от внешнего провайдера и сложность стабилизации результата.

Преимущества будущего подхода: модульность, возможность выбора провайдера, масштабирование набора reviewer. Недостатки: дополнительная стадия развития и необходимость будущей интеграции контекстного конвейера.

## 18. MVP Scope
MVP должен решать одну основную задачу: анализ локального Git-репозитория для C#-кода и выдачу понятного отчёта в Web UI.

### Входит в MVP

| Входит | Не входит |
|---|---|
| Работа с локальным Git-репозиторием | AI Reviewer |
| Определение изменённых файлов | CLI |
| Анализ C# через Roslyn | Git Hooks |
| Статический анализ | Многоязычная поддержка |
| Отчёт в Web UI | Интеграция с GitHub/GitLab |
| Markdown-экспорт | Распределённая обработка |

Почему выбран именно такой scope: он даёт законченный и демонстрируемый результат при ограниченном времени разработки. Альтернативой было бы расширение MVP до уровня платформы, но это сделало бы релиз слишком рискованным.

Преимущества: управляемый объём, высокая вероятность завершения, понятная проверка результата. Недостатки: часть архитектурных возможностей реализуется только как задел.

## 19. Roadmap
Дальнейшее развитие проекта должно идти поэтапно, без разрушения уже принятой архитектуры.

| Этап | Содержание |
|---|---|
| MVP | Анализ C#-репозиториев и генерация отчётов |
| Phase 2 | Подключение AI-модуля как отдельного слоя |
| Phase 3 | CLI и Git Hooks |
| Phase 4 | Новые языки через ILanguageAnalyzer |
| Phase 5 | Интеграции с GitHub/GitLab |

Почему выбран именно такой порядок: сначала нужно зафиксировать базовый сценарий и только потом добавлять дополнительные способы запуска и новые каналы интеграции. Альтернатива — параллельное развитие всех направлений — слишком рискованна для небольшой команды.

Преимущества: ясная последовательность, снижение неопределённости, возможность проверять ценность каждого этапа отдельно. Недостатки: некоторые будущие сценарии появятся не сразу.

## 20. Risks
Основные риски проекта связаны не только с техникой, но и с ограничениями MVP.

| Риск | Вероятное влияние | Митигирование |
|---|---|---|
| Ограниченное время | Неполная реализация второстепенных функций | Жёсткий MVP scope |
| Сложность анализа больших репозиториев | Рост времени обработки | Ограничение объёма входных данных |
| Семантические ошибки Roslyn-контекста | Неполный анализ зависимостей | Фокус на C# и валидном solution |
| Рост сложности при добавлении AI | Увеличение числа стадий pipeline | Отдельный AI-модуль |
| Избыточная абстракция | Усложнение разработки | Clean Architecture Lite вместо полной версии |

Почему выбран именно такой набор мер: они прямо соответствуют наиболее вероятным проблемам раннего MVP. Альтернативой было бы проектировать сразу “на вырост” с большим запасом сложности, но это противоречит срокам и команде из одного разработчика.

Преимущества: риски известны заранее и контролируются. Недостатки: часть проблем проявится только при более широком использовании системы.

## 21. Architectural Decisions
Ниже зафиксированы ключевые архитектурные решения, принятые на текущий момент.

| Решение | Почему выбрано | Альтернативы | Почему альтернативы не подходят для MVP |
|---|---|---|---|
| ASP.NET Core 9 | Единый современный backend-стек | Node.js, Python, Java | Не соответствуют выбранной .NET-ориентации и увеличивают разнотехнологичность |
| Blazor Server | Быстрый UI на C# без отдельного SPA | React, Angular, Vue | Добавили бы отдельный frontend stack и усложнили бы поставку |
| PostgreSQL | Надёжная реляционная БД | SQLite, MongoDB | SQLite хуже для роста и истории, MongoDB избыточна для структурированных данных |
| EF Core Code First | Быстрая эволюция модели | Dapper, raw SQL | Хуже для скорости разработки и сопровождения на MVP |
| LibGit2Sharp | Нативный Git API в .NET | CLI git calls | CLI-зависимость и сложнее обработка ошибок |
| Roslyn | Семантический анализ C# | Regex, сторонние парсеры | Недостаточно точны для кода C# |
| Clean Architecture Lite | Баланс простоты и изоляции | Полная Clean Architecture, модульный монолит с большим числом слоёв | Избыточны для двухнедельного MVP |
| Repository Pattern | Изоляция данных от логики | Прямой DbContext везде | Сильнее связывает приложение с EF Core |
| Dependency Injection | Композиция и тестируемость | Service Locator, ручная сборка | Ухудшает прозрачность и тестируемость |

Ключевой смысл этих решений: система должна быть достаточно строгой, чтобы пережить эволюцию, но достаточно простой, чтобы быть реализованной в ограниченный срок. Каждое решение принято в пользу предсказуемости и поддержки, а не в пользу максимальной теоретической гибкости.

## 22. Future Improvements
После MVP архитектура должна допускать следующие направления развития без разрушения существующего ядра.

| Направление | Описание |
|---|---|
| AI Review | Подключение специализированных AI reviewer как отдельного модуля |
| CLI | Отдельный интерфейс запуска анализа из командной строки |
| Git Hooks | Автоматический запуск перед commit/push |
| Multi-language support | Подключение новых ILanguageAnalyzer |
| GitHub/GitLab integration | Встраивание результатов в pull/merge request процессы |
| Report formats | Добавление HTML/PDF помимо Markdown |
| Background processing | Вынос тяжёлых задач в фоновый механизм |

Почему это важно: MVP должен быть не тупиковым продуктом, а устойчивой основой для дальнейшего развития. Альтернатива — закрытая одноразовая реализация — проще в краткосрочной перспективе, но создаёт высокий риск переписывания.

Преимущества: управляемая эволюция, возможность поэтапно расширять ценность продукта, сохранение уже принятых границ. Недостатки: потребуется дисциплина, чтобы новые функции не разрушали минималистичную основу.
